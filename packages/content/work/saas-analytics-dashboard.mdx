## The Problem

When I first spoke with the founding team, they described their situation clearly: their product was fundamentally good, but customers couldn't see the value it was creating. The analytics layer — which was supposed to be the product's killer feature — was essentially a data export button.

Users would log in, click "Export," wait for a CSV, open Excel, build pivot tables, and try to understand what was happening in their business. This added 30-60 minutes of manual work every time a customer wanted to answer a business question.

The churn signal was obvious in retrospect: customers weren't churning because the core product didn't work. They were churning because the friction to extract value was too high. The CEO described a customer call where a user said, "We love what the product does, but looking at the data is painful, so we just stopped checking it."

That is a product death sentence.

## The Architecture Decision

Before writing a line of code, I spent two days on architecture. The key constraints were:

1. **Multi-tenant from day one.** The platform serves multiple SaaS customers, each with their own data. Row-level security in PostgreSQL with a tenant-aware Prisma setup.
2. **Near-real-time, not real-time.** True real-time (WebSocket streaming) was over-engineered for the use case. What customers needed was data that was current within 5 minutes. This simplified the architecture significantly.
3. **Stripe billing is non-negotiable.** Different tiers get different data retention and export limits. This had to be baked into the data model, not bolted on.

The final architecture:

```
Event Ingestion API (Edge)
  ↓
Redis Queue (BullMQ)
  ↓
Worker Processes (Node.js)
  ↓
PostgreSQL (Prisma, row-level multi-tenant)
  ↓
Next.js App Router (RSC + Client Islands)
  ↓
Vercel Edge Network
```

The decision to use BullMQ for queuing rather than direct database writes was critical. It decoupled the ingestion rate from the processing rate, preventing database connection exhaustion during traffic spikes.

## The Tenant Data Model

The Prisma schema for multi-tenancy was designed so that RLS (Row Level Security) could be enforced at the PostgreSQL level, not just the application level:

```prisma
model Organization {
  id        String   @id @default(cuid())
  name      String
  slug      String   @unique
  planId    String
  createdAt DateTime @default(now())
  events    Event[]
  dashboards Dashboard[]
  apiKeys   ApiKey[]
  
  @@index([slug])
}

model Event {
  id             String       @id @default(cuid())
  organizationId String
  organization   Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)
  type           String
  properties     Json
  userId         String?
  sessionId      String?
  createdAt      DateTime     @default(now())
  
  @@index([organizationId, createdAt])
  @@index([organizationId, type, createdAt])
  @@index([organizationId, userId])
}
```

Every query in the application goes through a tenant-aware helper that enforces the organization filter:

```typescript
export function tenantPrisma(organizationId: string) {
  return {
    events: {
      findMany: (args: Prisma.EventFindManyArgs) =>
        prisma.event.findMany({
          ...args,
          where: { ...args.where, organizationId },
        }),
      // ... other methods
    },
  };
}
```

## The Dashboard UI

The dashboard UI was built with Next.js App Router using Server Components for all data fetching and Client Components only for interactive elements like date pickers, filter controls, and chart zoom.

The most technically interesting part was the chart system. We used a custom-built aggregation layer on top of PostgreSQL's window functions to compute time-series data efficiently:

```sql
SELECT
  date_trunc('day', created_at) as date,
  type as event_type,
  COUNT(*) as count,
  COUNT(DISTINCT user_id) as unique_users
FROM events
WHERE
  organization_id = $1
  AND created_at >= $2
  AND created_at <= $3
GROUP BY 1, 2
ORDER BY 1 ASC;
```

This single query, with proper indexing, returned a full month of daily event data for any organization in under 8ms on our PostgreSQL instance.

## Stripe Integration

Billing was tier-based with the following plan structure:

- **Starter** ($49/mo): 90 days data retention, 3 dashboards, no custom events
- **Growth** ($149/mo): 1 year retention, unlimited dashboards, custom events
- **Scale** ($399/mo): Unlimited retention, priority ingestion, API access

The Stripe webhook handler was the most critical piece of infrastructure — any failure here meant a customer could lose access to their data or be charged incorrectly:

```typescript
// app/api/webhooks/stripe/route.ts
export async function POST(req: Request) {
  const sig = req.headers.get("stripe-signature");
  const body = await req.text();
  
  let event: Stripe.Event;
  try {
    event = stripe.webhooks.constructEvent(
      body,
      sig!,
      process.env.STRIPE_WEBHOOK_SECRET!
    );
  } catch {
    return new Response("Invalid signature", { status: 400 });
  }
  
  // Idempotent handler — safe to retry
  await handleStripeEvent(event);
  return new Response("OK", { status: 200 });
}
```

The handler was fully idempotent using a `processedWebhooks` table keyed on Stripe's event ID — critical for reliability since Stripe can deliver webhooks multiple times.
